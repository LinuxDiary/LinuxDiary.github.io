# 文件描述符

# 概述

对于内核而言，所有打开的文件都是通过文件描述符引用的，文件描述符在形式上是一个非负整数。实际上，它是一个索引值，指向内核为每一个进程所维护的该进程打开文件的记录表。当程序打开一个现有文件或者创建一个新文件时，内核向进程返回一个文件描述符。每个Unix进程（除了可能的守护进程）应均有三个标准的 POSIX 文件描述符，对应于三个标准流：0（stdin）、1（stdout）、2（stderr），因此每个进程的文件描述符都是从3开始。

# 打开和分配的fd数量

# 系统级

主要涉及三个文件 file-max、file-nr、nr_open ，位于 /proc/sys/fs ,是动态的，以下是官方对三个文件的解释。 file-max 记录的是系统总共可打开的文件描述符数，是系统启动时根据硬件配置赋予的建议值，且可在 sysctl.conf 中进行定义。 file-nr 有三个值，第一个代表系统中已打开的文件描述符数（系统中总的 fd 打开数，包含进程的 fd 数），第二个值总是 0 ，第三个值同 file-max 。 nr_open 代表一个进程可分配的最大 fd 数，一般为 1024*1024=1048576 ，通常够用，无需修改。

==============================================================

file-max & file-nr:

The value in file-max denotes the maximum number of file- handles that the Linux kernel will allocate. When you get lots of error messages about running out of file handles, you might want to increase this limit.

Historically,the kernel was able to allocate file handles dynamically, but not to free them again. The three values in file-nr denote the number of allocated file handles, the number of allocated but unused file handles, and the maximum number of file handles. Linux 2.6 always reports 0 as the number of free file handles -- this is not an error, it just means that the number of allocated file handles exactly matches the number of used file handles.

Attempts to allocate more file descriptors than file-max are reported with printk, look for "VFS: file-max limit reached".

nr_open: This denotes the maximum number of file-handles a process can allocate. Default value is 1024*1024 (1048576) which should be enough for most machines. Actual limit depends on RLIMIT_NOFILE resource limit.

==============================================================

# 进程级

查看进程级 fd 数可通过 lsof 命令，但由于 lsof 命令包含了并非以 fd 形式打开的文件，如 mem mmap 等，直接使用 lsof 获取的 fd 数将远大于真实的 fd 数，因为有很多文件可能会被多进程和多线程打开多次，被重复打开的文件很可能没有 fd ，而 lsof 会将这些全部列出来，因此用如下办法获取较为真实的进程级 fd 数量

$ find /proc -print | grep -P "/proc/\d+/fd/"|wc -l
$ lsof -n |awk '{print $4}'| egrep '^[0-9]+' |wc -l

查看每个进程打开的 fd 数

$ ls -l /proc/<pid>/fd |wc -l

# FDSize

FDSize 是系统给每个进程分配的fd数，如果进程打开的fd超过64，64位系统以64为基数动态递增，32位系统同理。 FDSize 通过 cat /proc/<pid>/status 获得，受 /proc/<pid>/limits 中的 open files 限制。

# 调整进程的fd数量上限

如未指定，系统默认情况下非 root 用户的 open files 的 soft 上限是 1024 ， max user processes 的 soft 上限是 4096 （ CentOS7 在 /etc/security/limits.d/20-nproc.conf 中指定）。系统的 services 不受 limits.conf 的限制，这个文件仅仅限制通过 pam 认证登陆的系统用户。这是 CentOS7 与 CentOS6 及之前的版本所不同的地方， CentOS6 可通过直接修改 limits.conf 开控制 limits 值，而基于 systemd 的 CentOS7 不行。以下是 limits.conf 的描述

# /etc/security/limits.conf
#
#This file sets the resource limits for the users logged in via PAM.
#It does not affect resource limits of the system services.
#

基于 systemd 的 services 的 open files 上限由 /etc/systemd/system.conf 和服务本身的 .service 文件配置，且优先级 .service > system.conf 。其中 system.conf 配置的是默认的 limits 值，修改后需重启系统。

.service 文件中以 LimitNOFILE = 和 LimitNPROC = 来指定该服务的 open files 和 processes 的上限，修改后重启服务可通过 /proc/<pid>/limits 查看生效。

system.conf 文件中以 DefaultLimitNOFILE= 和 DefaultLimitNPROC= 来设置默认的 open files和processes 的 limits 值。设置后，如果某系统服务未指定 LimitNOFILE = 和 LimitNPROC = ，将以 DefaultLimitNOFILE= 和 DefaultLimitNPROC= 的值作为默认上限，可通过 /proc/<pid>/limits 查看生效。

以mysql服务和用户为例

$ cat /etc/passwd |grep mysql
mysql:x:27:27:MariaDB Server:/var/lib/mysql:/sbin/nologin

mysql 为 MySQL 服务创建的不可登陆的系统用户，修改其 limits 为极低值

$ sudo vi /etc/security/limits.d/20-nproc.conf
mysql - nofile 2
root soft nproc unlimited

而之前在 mysqld.service 和 system.conf 中指定的 LimitNOFILE 值不变

$ sudo vi /usr/lib/systemd/system/mysqld.service
# Sets open_files_limit
LimitNOFILE = 10240
$ sudo vi /etc/systemd/system.conf
DefaultLimitNOFILE=20480

重启 mysqld.service

$ sudo systemctl daemon-reload
$ sudo systemctl restart mysqld

获取 mysqld 新的 pid

$ pidof mysqld
1040

查看 mysqld 的 limits 对比另一进程的 limits

$ cat /proc/1040/limits
Max open files 10240 10240 files
$ cat /proc/1198/limits
Max open files 20480 20480 files

由此看出， limits.conf 和 limits.d/*.conf 只能影响登陆 shell 的用户的 limits ，而不能影响基于 systemd 的系统服务的 limits ，且 .service 的优先级大于 system.conf
