# 主从复制

  • 系统环境：CentOS 7
  • 软件版本：MySQL 5.6
  • Master：192.168.2.13
  • Slave：192.168.2.14

# 基于LOG POS

# Master上的主要配置

log_bin = /home/mysql/u01/mysql/binlog/binlog
binlog_cache_size = 32768
binlog_format = ROW
expire_logs_days = 14
log_slave_updates = ON
sync_binlog = 1

server-id = 101

innodb_flush_log_at_trx_commit = 1

# Master上创建同步用户

GRANT REPLICATION SLAVE ON *.* to rep_user@'%' IDENTIFIED BY 'PASSWORD';
FLUSH PRIVIELEGS;

# mysqldump导出Master数据库

整库导出 Master ，由于即有 MyISAM 表也有 InnoDB 表，因此在导出过程中需要全局锁表(--lock-all-tables)，记录 LOG POS (--master-data=2)，并刷新二进制日志(--flush-logs)

mysqldump --user=root --password --triggers --events --routines --flush-logs --lock-all-tables --master-data=2 --all-databases > all_data.sql

如果只需要同步某一个 InnoDB 的库，且 binlog_format = ROW ，那么可以使用 binlog-do-db = db1 在 Master 上指定只记录 db1 的 binlog ，同步时 Slave 也只会读取到这些 binlog 。在这种情况下，使用下面命令导出 db1 数据库，在 Slave 上导入时如果没有 db1 ，就手动创建一个

mysqldump --user=root --password --triggers --events --routines --flush-logs --single-transaction --opt --master-data=2 --databases db1 > db1.sql

# Slave上的主要配置

binlog 相关设置与 Master 基本一致，但是 server-id 必须唯一，区别于 Master ， Slave 上除了基本的 relay_log 相关配置之外， 还设置了一个 skip-slave-start 选项，作用是 MySQL 启动时不自动开启 slave ，并将 Slave 设置成只读模式，非 super user 不能修改 Slave 中的数据。

log_bin = /home/mysql/u01/mysql-5.6.39/binlog/binlog
binlog_cache_size = 32768
binlog_format = ROW
expire_logs_days = 14
log_slave_updates = ON
sync_binlog = 1

server-id = 102

innodb_flush_log_at_trx_commit = 1

relay_log = /home/mysql/u01/mysql-5.6.39/log/relaylog
relay_log_index = /home/mysql/u01/mysql-5.6.39/log/relay.index
relay_log_info_file = /home/mysql/u01/mysql-5.6.39/log/relay-log.info

slave_parallel_workers = 4
slave_load_tmpdir = /home/mysql/u01/mysql-5.6.39/tmp
slave_skip_errors = OFF
skip-slave-start
read_only = ON

sync_master_info = 1
sync_relay_log =1
sync_relay_log_info = 1

sync_master_info sync_relay_log sync_relay_log_info 三个选项可以最大限度的防止因 Slave 服务器崩溃导致 master_info 中继日志信息不完整，但是相应的会增加一些系统开销

slave_parallel_workers 是多个线程复制选项，可以指定多个 salve 工作线程，最大 1024 ，最小和默认是 0 ，该特性是 MySQL 5.6.3 版本以后加入的

TIP

slave_parallel_workers 选项在 5.6 版本中只适用于多个 schema 的场景，可以实现多个库并行复制，对于一库多表的场景，这个选项并不能发挥作用。但是在 5.7 版本中引入了基于组提交的并行复制(Enhanced Multi-threaded Slaves)，可以通过设置 slave_parallel_type＝'LOGICAL_CLOCK' 实现基于组提交的并行复制，详情参考：https://dev.mysql.com/doc/refman/5.7/en/replication-options-slave.html#sysvar_slave_parallel_type

另外，可以根据实际情况设置几个可以忽略的同步错误代码，例如 slave_skip_errors = 1032,1062,1007,1008

# 导入Master上的数据库

mysql -uroot -p < all_data.sql

如果是针对某一个库的同步，则只导入一个库的数据

mysql -uroot -p db1 < db1.sql

如果数据库比较大，可以使用 screen 工具将数据导入任务放在后台运行。

# Slave上开启同步

在数据导入成功后，登录到 Slave 上，清除所有二进制日志和中继日志，有必要时先备份这些日志

RESET MASTER;
RESET SLAVE ALL;

CHANGE MASTER 到前面 all_data.sql 中记录的 LOG POS 信息，然后启动 salve

CHANGE MASTER TO MASTER_HOST='192.168.2.13',MASTER_LOG_FILE='mysql-bin.000020', MASTER_LOG_POS=107;
START SLAVE USER='rep_user' PASSWORD='PASSWORD';

检查同步状态， IO 和 SQL 线程都是 Yes ，表示同步已正常，多刷新几次观察日志同步的位置变化

# 基于GTID

# GTID简介

GTID(Global Transaction ID) 是对于一个已提交事务的编号，并且是一个全局唯一的编号。 GTID 实际上是由 UUID+TID 组成的。其中 UUID 是一个 MySQL 实例的唯一标识。 TID 代表了该实例上已经提交的事务编号，并且随着事务提交单调递增。

# 配置方法

分别在 Master 和 Slave 上开启 gitd 模式，在传统主从复制配置的基础上增加以下选项

gtid_mode=ON
log_slave_updates
enforce_gtid_consistency

# 开启slave

启用 gtid 后，在导出的 SQL 文件中会包含以下语句，表示事务编号 1-6127131 已经执行过，在导入 Master 上导出的数据之后，同步将从事务 ID 6127132 开始。

SET @@GLOBAL.GTID_PURGED='ca9fd6a6-1a2e-11e8-8842-525400642e95:1-6127131';

在 Slave 上 CHANGE MASTER ，然后开启 slave ，在此之前先清除 binlog 和 relay-log

RESET MASTER;
RESET SLAVE ALL;
CHANGE MASTER TO MASTER_HOST='192.168.2.13', MASTER_PORT=3006, MASTER_AUTO_POSITION=1;
START SLAVE USER='rep_user' PASSWORD='PASSWORD';

当然也可以直接 CHANGE MASTER ， Slave 会自动从事务 ID 1 开始同步，过程中可能会出现同步报错，例如表已存在的错误，可以视情况是否忽略。开启成功后，检查 slave 状态是否正常。
