# 迁移案例

# 境数据库现状

之前生产环境 MySQL 由研发人员使用和管理，单库单服务器运行。接手后发现存在以下问题：

  • 业务程序使用 root 用户连接数据库；
  • binlog 没有开启，慢查询日志没有开启；
  • 同一库中的表，使用了两种存储引擎： MyISAM 和 InnoDB ；
  • max_connections 和 open_files_limit 都是默认的 151 和 1024 ；

其他影响较小的问题未列出

# 整改目标

  • 规范用户权限，限制业务应用连接数据库的权限(增删改查)；
  • 开启 binlog 和慢查询日志；
  • 配置主从同步，使用 GTID ；
  • 同一表的存储引擎为 InnoDB ；
  • 配置 MySQL 的默认字符集和校验集；
  • 优化连接数和打开文件数；
  • 优化 InnoDB 相关参数；
  • 使用 XtraBackup 定时备份数据库

# 重新规划

Master

  • IP：192.168.1.11 (原库所在服务器)
  • 硬件配置：E5-2620 12核*2/32G/1800G Raid5

Slave

  • IP：192.168.1.13
  • 硬件配置：E5-2620 12核/16G/1400G Raid5

原库当前运行在 Master 上，计划在 Master 上新建一个 MySQL 的专用用户，用来运行新的 MySQL 服务，使用 3307 端口，与原库区分开来，部署过程中原库不关闭。

# 转储数据

总共有 2 个库需要转储， zhcj 和 data_sync ，数据基本都在 zhcj 中， data_sync 是辅助的库，体量很小。

在原库服务器上使用 mysqldump 工具导出原数据库，由于存储引擎不统一，只能进行全局锁表的导出方式

mysqldump -uroot -p --socket=/var/lib/mysql/mysql.sock --events --lock-all-tables --databases zhcj data_sync > all.sql

在导 zhcj 库时出现问题，由于原数据库默认使用了 latin1 编码，在导出 zhcj 库之后，中文乱码，即便是使用了 --default-character-set 选项也无济于事，但是导出表结构却是正常，整库导出就乱码，于是分表导出，然后批量修改引擎。

screen -S dump

创建 dump.sh 脚本

#!/bin/bash
mysql -uroot -p'PASSWORD' --socket=/var/lib/mysql/mysql.sock -e "show tables from zhcj\G" | grep Tables_in_zhcj | awk '{print $2}' > table_list
for table in `cat table_list`
do
mysqldump -uroot -p'PASSWORD' --socket=/var/lib/mysql/mysql.sock --events --lock-all-tables zhcj $table -r table_out/$table.sql
done >> ~/dump.log 2>&1

cd table_out
for table in `ls`
do
sed -i 's/ENGINE=MyISAM/ENGINE=InnoDB/g' $table
done >> ~/sed.log 2>&1

在导入数据到新 Master 库之前，临时配置一下 innodb_flush_log_at_trx_commit=0 ，这样导入数据时速度会稍微快一点。这样的配置，假如出现故障，可能会丢失1s的数据。但是由于当前已确保服务器的正常运行，因此暂时配置该选项。

数据导出并修改好存储引擎之后，将数据导入到新 Master 数据库中

先导入表结构，原因是库中存在关联的表，直接导入所有数据可能会因为有些表还未创建，导致错误提示：找不到xxx表

screen -S mysql_in
sudo mysqldump -uroot -p'PASSWORD' --socket=/var/lib/mysql/mysql.sock --events --lock-all-tables --no-data zhcj  -r zhcj_s.sql
sudo sed -i 's/ENGINE=MyISAM/ENGINE=InnoDB/g' zhcj_s.sql
mysql --socket=/home/mysql/u01/mysql-5.6.39/run/mysql.sock -uroot -p'PASSWORD' zhcj < zhcj_s.sql

再执行批量导入

cd table_out
for table in `ls`
do
mysql --socket=/home/mysql/u01/mysql-5.6.39/run/mysql.sock -uroot -p'linuxd.cc' zhcj < $table
done >> ~/run.log 2>&1

在数据转储的整个过程中，由于原库没有配置 binlog ，无法做增量，势必丢失一些新写入的数据，无法避免。

zhcj 转储完成之后，导入 data_sync ，整个转储就完成了，如果 Master 中还没有 data_sync ，就手动创建

mysql --socket=/home/mysql/u01/mysql-5.6.39/run/mysql.sock -uroot -p'PASSWORD' data_sync < data_sync.sql

# 创建专用MySQL用户

在 Master 新库上创建应用用户、同步用户、备份用户

grant SELECT, UPDATE, DELETE, INSERT on zhcj.* to zhcj_user@'%' identified by 'PASSWORD';
grant SELECT, UPDATE, DELETE, INSERT on data_sync.* to zhcj_user@'%' identified by 'PASSWORD';
grant REPLICATION SLAVE on *.* to rep_user@'%' identified by 'PASSWORD';
grant CREATE TABLESPACE, RELOAD, PROCESS, SUPER, LOCK TABLES, REPLICATION CLIENT on *.* to backup@'localhost' identified by 'PASSWORD';
FLUSH PRIVIELEGS;

创建好之后，在应用系统中配置新的数据库连接信息，测试应用系统

# 配置Slave

确定 Master 运行正常之后，清空 Slave 的 binlog 和 relay-log ，如果之前 Slave 上有同步报错，这里可以一并清除

RESET MASTER;
RESET SLAVE ALL;

开启 slave ，在 CHANGE MASTER 时不包含用户密码，而是在 START SLAVE 时使用密码，注意用户密码信息就不会保存在master.info 文件中，所以在设置 MySQL 配置文件时使用了 skip-slave-start 选项

CHANGE MASTER TO MASTER_HOST='192.168.1.11', MASTER_PORT=3307, MASTER_AUTO_POSITION=1;
START SLAVE USER='rep_user' PASSWORD='PASSWORD';

# 定时物理备份Master

每天凌晨 1 点备份数据库， 2 点检查超过 30 天的备份并删除

使用 XtraBackup 对数据库执行热备任务，创建日志目录 mkdir /home/mysql/backup_log 后，然后创建备份脚本 full_backup.sh

#!/bin/bash

backup_dir=/home/mysql/master_backup/$(date +%Y-%m-%d)
if [ -d $backup_dir ];then
   backup_dir=/home/mysql/master_backup/$(date +%Y-%m-%d)_1
fi

mkdir -p $backup_dir
cd $backup_dir

innobackupex --defaults-file=/home/mysql/u01/mysql/etc/my.cnf \
    --no-timestamp \
    --use-memory=1G \
    --parallel=4 \
    --user=backup \
    --password='PASSWORD' ./ > \
    ~/backup_log/backup-progress-$(date +%Y-%m-%d).log 2>&1 &

定时任务

$ sudo crontab -l
0 1 * * * /bin/bash /home/mysql/full_backup.sh
0 2 * * * /usr/bin/find /home/mysql/master_backup -type d -mtime +30 -exec rm -rf {} \;

# rsync定时同步

将备份的数据库文件定时同步到其他服务器上

  • rsync服务器：192.168.1.13（Slave）
  • rsync客户端：192.168.1.11（Master）

部署rsync服务端

编辑配置文件 /etc/rsyncd.conf

uid = mysql   #文件传输到本地后的属主
gid = mysql   #文件传输到本地后的属组
max connections = 36000
use chroot = no
log file = /var/log/rsyncd.log
pid file = /var/run/rsyncd.pid
lock file = /var/run/rsyncd.lock
dont compress   = *.gz *.tgz *.zip *.z *.Z *.rpm *.deb *.bz2

[backup_dir]   #模块id
path = /home/mysql/backup_dir   #模块对应的路径
ignore errors   
read only = false
write only = false
hosts allow = 192.168.1.0/24
hosts deny = 0.0.0.0/32
auth users = rsync_user   #认证用户
secrets file = /etc/rsyncd.pass   #认证用户密码，权限必须是600

用户密码文件

$ sudo vi /etc/rsyncd.pass
rsync_user:123456
$ sudo chmod 600 /etc/rsyncd.pass

启动rsync服务

systemctl start rsyncd.service
systemctl enable rsyncd.service

配置 SELinux

setsebool -P rsync_full_access on

创建模块目录

su - mysql
mkdir -p /home/mysql/backup_dir

客户端测试rsync连接

sudo bash -c 'echo "123456" >> /etc/rsync.pass'
sudo chmod 600 /etc/rsync.pass
sudo rsync --list-only rsync://rsync_user@192.168.1.13:873/backup_dir  --password-file=/etc/rsync.pass
drwxrwxr-x              6 2018/05/10 09:45:06 .

rsync 服务搭建成功后，在 Master 上添加再一个计划任务

$ sudo crontab -l
0 1 * * * /bin/bash /home/mysql/full_backup.sh
0 3 * * * /usr/bin/find /home/mysql/master_backup -type d -mtime +30 -exec rm -rf {} \;
0 4 * * * /usr/bin/rsync -az --delete /home/mysql/master_backup rsync://rsync_user@192.168.1.13:873/backup_dir --password-file=/etc/rsyncd.pass
