# InnoDB特性

# ACID

ACID 是事务四要素：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）。是事务型数据库必须支持的四个特性， InnoDB 是典型的事务型存储引擎。

原子性

一个事务中的所有操作，要么全部执行成功，要么全部失败。事务中只要有一个操作执行失败，整个事务就会回滚到事务开始前的状态。

一致性

事务执行前后，数据库总体保持一致，写入的数据符合预设规则，例如银行转账，从一个账户转到另一个账户的情况下，两个账户的总金额保持不变。

隔离性

允许多个并发事务对数据进行读写操作，根据不同的隔离级别，呈现不同的效果。 InnoDB 中事务隔离级别有四种：读未提交（Read uncommitted）、读提交（read committed）、可重复读（repeatable read）和串行化（Serializable）

持久性

事务执行成功并提交之后，对数据的修改是永久性的，永久的保存在数据文件中。

# 事务隔离级别

READ-UNCOMMITTED

非提交读。例如事务 A 和 B ，在事务 B 中修改了一条数据，在事务 A 中可以 select 查询到事务 B 中修改后但未提交的数据。该级别允许脏读、非重复读、幻读。

READ-COMMITTED

提交读。在事务 B 中修改了一条数据，在事务 A 中可以查询到事务 B 中修改并提交后的数据。该级别允许非重复读、幻读。

REPEATABLE-READ

重复读。在事务B中修改了数据并提交之后，事务 A 中仍然查询到的是修改前的数据(就是重复读)。这就保证了在同一个事务中数据的一致性， A 事务不受 B 事务的影响。该级别默认情况下是只允许幻读，但是在 InnoDB 的标准中，脏读、非重复读、幻读都不允许。

SERIALIZABLE

与 REPEATABLE-READ 类似，都不允许脏读、非重复读、幻读，但是性能相对较差。

# MVCC

MVCC (Multiversion Concurrency Control) ，多版本并发控制，在 InnoDB 表中实现于事务隔离级别 REPEATABLE-READ 中，在某个事务中，存储某行记录的多个快照，不同的事务之间相互没有读写干扰。可重复读的隔离级别下，就算其他事务对同一行记录执行的 update 操作，在当前事务中查看该行记录的结果始终一致，也就是查看了该行记录早期的一个快照版本。

# 锁

InnoDB 在读数据时，实际上读到的是行记录的快照，不同的隔离级别下，读取的快照有所不同，这种特性称为一致性非锁定读，发生在 READ-UNCOMMITTED 和 REPEATABLE-READ 两种级别之下。在 READ-UNCOMMITTED 级别下，读取的数据是最新的快照，而在 REPEATABLE-READ 级别下，读取的数据则是事务开始时，最开始生成的快照，这就符合了对应隔离级别所产生的效果。

与之对应的是一致性锁定读，用于用户显式的对数据库的读取操作进行加锁，例如 InnoDB 对 SELECT 语句支持两种锁定读操作

SELECT ... FOR UPDATE
SELECT ... LOCK IN SHARE MODE

FOR UPDATE 表示加 X 锁， LOCK IN SHARE MODE 表示加 S 锁，这两种显式加锁的方式只能在一个启动的事务中执行，事务提交或退出后锁自动失效。

# 锁的类型

  • 共享锁：就是 S 锁，也叫读锁，事务在读取数据时可能存在的锁
  • 排他锁：就是 X 锁，也叫写锁，事务在删除或更新一行数据时的锁

两种锁的兼容性列表

X S
X 不兼容 不兼容
S 不兼容 兼容

# 死锁

死锁的产生是由于多个事务争用锁资源导致，如下是一个典型死锁示例

时间 事务A 事务B
1 BEGIN;
2 SELECT * FROM T WHERE a=1 FOR UPDATE; BEGIN;
3 SELECT * FROM T WHERE a=2 FOR UPDATE;
4 SELECT * FROM T WHERE a=2 FOR UPDATE;
5 SELECT * FROM T WHERE a=1 FOR UPDATE;
