# 参数

# binlog_format

# STATEMENT

STATEMENT-based replication(SBR) 基于 SQL 语句的复制模式。 binlog 只记录对数据库执行了更改的 SQL 语句，在 Slave 端能够正确的执行这些语句。对于某些函数(例如 sleep() )无法被复制，可能导致 Master 和 Slave 数据不一致的情况。因为只记录了 SQL 语句，所以日志的大小会比较小。

# ROW

ROW-based replication(RBR) 基于行的复制模式。 binlog 会清楚的记录表中每一行发生的变化，对于某些函数执行的结果也一样记录，但不记录对数据库更改时执行的 SQL 语句和上下文关系。因为要记录每一行的改变细节，所以 ROW 模式可能会产生大量的日志。

# MIXED

MIXED-based replication(MBR) 以上两种模式的混合使用，MySQL 5.6 版本中默认使用这种格式，一般的复制使用 STATEMENT 模式记录 binlog ，大多数情况下， MIXED 与 STATEMENT 的效果一样。对于 STATEMENT 模式无法记录的操作使用 ROW 模式记录 binlog ， MySQL 会根据执行的 SQL 语句选择日志保存方式。

# binlog-do-db

binlog-do-db、binlog-ignore-db、replicate-do-db、replicate-ignore-db 是同类的四个选项，在 MySQL 主从复制的场景中使用。 binlog-do-db 基本含义是指定要记录或忽略不记录的 binlog 的数据库名，指定要复制或不复制的数据库名。

# binlog-do-db生效机制链接

# 情况1

  • binlog_format = ROW
  • binlog-do-db = db1
  • mysql> USE db1; 或 mysql> USE db2;

在 ROW 模式下，所有的 DDL/DML 操作都只对 binlog-do-db = db1 起作用，即便 USE db2 之后，也不会对 db2 产生任何影响。换言之， ROW 模式下对选项 binlog-do-db 的 binlog 记录，只针对于 binlog-do-db 所指定的库。

# 情况2

  • binlog_format = STATEMENT
  • binlog-do-db = db1
  • mysql> USE db1;

指定的是库 db1 ，只有在 USE db1 之后，默认库是 db1 的情况下，所执行的所有操作(包括 DDL/DML )都会记录 binlog ，即便执行的是针对 db2 的操作。换言之， STATEMENT 模式下对选项 binlog-do-db 的 binlog 记录，完全依赖于默认数据库(即 USE dbname )。

# 情况3

  • binlog_format = MIXED
  • binlog-do-db = db1

在多数情况下， MIXED 和 STATEMENT 效果一样，只有在 STATEMENT 模式无法记录的操作才会切换到 ROW 模式进行记录，在对应模式下的记录 binlog 的规则同上。

# 情况4

  • binlog_format = STATEMENT 或 ROW 或 MIXED
  • binlog-do-db = db1

与前面 3 种情况不同，在使用 DCL 语句( GRANT、REVOKE 等授权语句)的时候，不适用于上述规则。例如在执行 GRANT 授权语句时，只有在执行 USE db1 之后， GRANT 的操作才会被 Slave 复制，在其他库的时候， GRANT 的操作不会被 Slave 复制。

# nteractive_timeout和wait_timeout

  • interactive_timeout : 交互式连接的超时时间，默认是28800(8小时)
  • wait_timeout : 超时等待时间，默认是28800(8小时)

# 生效机制

交互式连接

典型的是 MySQL 客户端，实际起作用的参数是 wait_timeout ，这个参数决定了连接的超时时间。在交互式连接中， wait_timeout 的值可以由两种方式获得

set session wait_timeout =100; 只生效与当前连接

set global interactive_timeout =100; 设置之后，下一个交互式 session 的 wait_timeout 将继承 interactive_timeout 的值

非交互式连接

一般是程序与 MySQL 的连接，例如 PHP 、 Python 等，此时 wait_timeout 将继承自全局的 wait_timeout 值，也就是说 set global wait_timeout =100; 将决定下一个非交互式连接的超时时间

# 总结

MySQL 连接的超时等待时间(无论是交互式还是非交互式)，实际是由当前连接的 wait_timeout 值所决定，而 MySQL 根据连接的类型决定 wait_timeout 继承哪一个值，全局的 interactive_timeout 还是 wait_timeout

一般这两个值的设定根据实际情况而定，例如 interactive_timeout =600 wait_timeout=300
