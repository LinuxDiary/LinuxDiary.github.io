# Memcached基础

# 安装

# 安装依赖

sudo yum install libevent-devel

也可以编译源码安装 libevent ，如果编译安装，则需要在安装 Memcached 时需要指定 libevent 路径，例如 --with-libevent=/usr/local/libevent

# 安装

mkdir -p ~/u01/memcached/src
cd ~/u01/memcached/src
wget http://www.memcached.org/files/memcached-1.5.10.tar.gz
tar xf memcached-1.5.10.tar.gz
cd memcached-1.5.10
./configure --prefix=/home/www/u01/memcached
make && make install

# 常用启动参数

启动方式

memcached -d -m 256m -c 200 -p 11211 -P memcached.pid

使用 -v/-vv/-vvv 选项开启日志，或者重定向到文件中

memcached -d -m 256m -c 200 -p 11211 -P memcached.pid -v >> mem.log 2>&1
  • -p: 指定服务的 TCP 端口号
  • -s: 使用 socket 方式启动，指定 socket 路径
  • -A: 启用 shutdown 命令，命令行客户端中可使用 shutdown 关闭 memcached 服务
  • -l: 指定监听的 IP
  • -d: 以守护进程方式运行
  • -u: 指定用户(root)
  • -m: 指定最大可用内存
  • -c: 指定最大并发连接数，默认 1024
  • -P: 指定 pid 文件路径
  • -f: 指定自增长因子，默认是 1.25
  • -n: 设置最小 slab size ，默认是 48 byte ，加上 64 位机器的 item 大小 48 byte ，共同构成 chunk 的默认初始大小 96 byte

# 常用命令

  • set 新增一个 item ， set key flags exTime length -> value ，可以直接插入，不管 key 是否存在
  • add 只有当 key 不存在时才能插入，add key flags exTime length -> value
  • replace 只能修改已存在的 key ，如果 key 不存在，则不执行操作
  • append 在 key 的 value 后面追加内容， length 是追加的长度
  • prepend 在 key 的 value 前面追加内容， length 是追加的长度
  • get 获取 key 的值
  • cas check and set ， 检查版本号是否正确，然后执行操作， cas key flags exTime length unique_cas_token -> value
  • gets 获取带有 unique_cas_token 的 item 值
  • delete 删除 key ，给 key 打上 delete 标签，并非真正删除
  • incr/decr 对 key 进行自增长或自减少， incr/decr key increment_value ，只能对无符号的整数执行操作
  • flush_all 清理缓存中的所有数据，危险操作，尽量不使用

# 常用状态信息

通过 stats 命令查看 Memcached 整体状态信息

命中率相关

  • cmd_get 总 get 次数 cmd_get=get_hits+get_misses
  • get_hits 命中次数
  • get_misses 未命中次数

命中率通过 get_hits/cmd_get 计算获得

数据量相关

  • curr_items 当前可用的 item 数量
  • total_items 从系统运行起的 item 总量
  • evictions LRU 释放的 item 总量，如果此项过高，说明已经没有 chunk 可用，原因可能是 chunk 大小分配不合理或内存用尽
  • bytes 当前所占用的存储空间
  • bytes_read 读取的总字节数
  • bytes_written 写入的总字节数

资源使用相关

  • rusage_user 进程累计的执行命令时间
  • rusage_system 进程累计使用系统 CPU 的时间

# stats相关其他命令

  • stats items : 各个 slab 中 item 的数目和存储时长
  • stats slabs : 显示各个 slab 的详细信息，包括 chunk 大小和数量等详细信息

# 调优

# Chunk的使用机制

Memcached 中使用 Chunk 的机制是优先使用 Slot 中的 Chunk ，其中 Slot 中是被删除数据或过期数据释放的 Chunk 。Slot 中用尽以后，再去 Slab 中申请新的 Chunk ，直到所有 Chunk 用尽，用尽之后会启用 LRU 机制，释放最近最少使用的 Chunk 空间， Memcached 并不会使用不匹配 item 大小的其他空闲 Chunk ，因此需要调整初始 Chunk 大小和自增长因子。

内存浪费的几种情况

  • Chunk 总是不能被存满
  • 热点数据无法命中位于中间位置的 Chunk
  • Slab 不能被 Page 整除
  • Page 不能被 Chunk 整除

Memcached 调优多数情况下是调整 Chunk 大小和 自增长因子，通过具体的业务情况，计算主要的 Chunk 分布情况，然后根据分布情况调整初始 Chunk 大小和自增长因子，例如

memcached -d -m 256m -n 32 -f 1.1

# 限制

  • 不提供持久化机制
  • 数据最多保存 30 天，无论 Memcached 进程是否终止
  • 不提供安全机制，一般配合防火墙使用
  • key 的最大长度为 250 byte
  • 单个 item 最大长度为 1m
  • 不提供冗余机制
