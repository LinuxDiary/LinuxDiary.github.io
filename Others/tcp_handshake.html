# TCP三次握手四次挥手

# OSI模型

# TCP特点

TCP 传输控制协议，最大的特点就是可靠性。 TCP 的连接需要经过三次握手之后建立，经过四次挥手后关闭连接。三次握手和四次挥手是保证 TCP 连接可靠性的重要过程。除此之外， TCP 还具有丢包重传的特点。 TCP 建立连接时是三次握手，而关闭连接时是四次挥手，因为是三次握手中第二次是发送了 2 个消息 SYN+ACK ，其实这两个消息也可以分开发，在实际环境中有时候建立连接实际上是四次握手。

# 三次握手

三次握手过程如下所示， Client 与 Server 是相对而言，并非指具体的客户端和服务端。

从具体数据包看三次握手过程

每一个数据包都包含一个 seq 序列号，具体的 seq 和 ack 变化过程如下

  • Client 发送 SYN 消息， seq=u
  • Server 发送 SYN+ACK 消息， seq=w,ack=u+1
  • Client 发送 ACK 消息， seq=u+1,ack=w+1

由此看出，每一个数据包的 ack 消息都与上一个数据包有所关联，这是 TCP 可靠性的一种体现。

在 TCP 连接建立之后，一端向另一端发送数据包后，会收到一个 ack 的回复以确定数据包已送达，如果在一定时间内没有收到 ack 回复，则发送端会认为数据包没有送达，这时会重新发送一次，直到收到 ack 回复。

  • Server 发送 FIN 消息， seq=w,ack=u+1
  • Client 发送 ACK 消息， seq=u+1,ack=w+1

# TCP状态

TCP 无论是在三次握手还是四次挥手过程中， Client 和 Server 端 (通过 netstat -antp 命令可查看某时刻的 TCP 状态) 都是从一个状态转换到另一个状态，直到双方都达到 established 或 closed 状态，常见的 TCP 状态如下

状态 说明
LISTEN 侦听状态。
SYN-SENT Client 在发送连接请求后等待匹配的连接请求。通过connect()函数向 Server 发出一个同步（SYNC）信号后进入此状态。
SYN-RECEIVED Server 已经收到并发送同步（SYNC）信号之后等待确认（ACK）请求。
ESTABLISHED 连接已建立，可以开始发送数据。
FIN-WAIT-1 Client 或 Server 主动调用 close() 函数发出 FIN 请求包，表示本方的数据发送全部结束，等待连接另一端的确认包或 FIN 请求包。
FIN-WAIT-2 Client 或 Server 主动关闭端在 FIN-WAIT-1 状态下收到确认包，进入等待远程 TCP 的连接终止请求的半关闭状态。这时可以接收数据，但不再发送数据。
CLOSE-WAIT Client 或 Server 被动关闭端接到 FIN 后，就发出 ACK 以回应 FIN 请求，并进入等待本地用户的连接终止请求的半关闭状态。这时可以发送数据，但不再接收数据。
CLOSING Client 或 Server 在发出 FIN 后，又收到对方发来的 FIN 后，进入等待对方对连接终止（FIN）的确认（ACK）的状态。
LAST-ACK Client 或 Server 被动关闭端全部数据发送完成之后，向主动关闭端发送 FIN ，进入等待确认包的状态。
TIME-WAIT Client 或 Server 主动关闭端接收到 FIN 后，就发送 ACK 包，等待足够时间以确保被动关闭端收到了终止请求的确认包。[按照RFC 793，一个连接可以在TIME-WAIT保证最大四分钟，即最大分段寿命（maximum segment lifetime）的2倍]
CLOSED 连接完全关闭。
